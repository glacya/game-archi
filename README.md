# 게임 설계서
이 README는 제작 중인 상업용 게임 프로젝트의 설계 전반을 기술합니다.

#### 프로젝트 개요
이 프로젝트는 턴제 기반, 로그라이트 장르의 게임을 제작합니다.  
Unity 6.3 기반으로 제작되었으며, Model-View-Controller 디자인 패턴을 채택해 구현하고 있습니다.


#### 게임 기획 개요
이 게임은 턴제 기반 로그라이트입니다. 최대 4 명의 아군을 골라 파티를 짜고, 그 파티로 던전을 들어가 던전을 공략하는 방식입니다. 던전은 여러 개의 방으로 이루어져 있고, 방들은 서로 연결되어 있습니다. 
방들을 전부 공략해 마지막 층까지 도달하여 보스를 처치하면 던전을 클리어하는 방식입니다.

일반적인 로그라이크와 다른 로그라이'트'이기 때문에 플레이어는 게임에 패배해도 모든 것을 잃지 않습니다. 마을에서 파티를 강화하고, 아이템을 모아둘 수 있습니다.


### 개발
#### MVC 디자인 패턴
MVC(Model-View-Controller) 패턴은 소프트웨어 개발에서 흔히 사용되는 패턴입니다. 
프로젝트의 구성 요소를 논리적 요소(Model)와 시청각적 요소(View), 그리고 그 둘을 중재하고 제어하는 역할을 맡는 요소(Controller)의 구조로 나눠놓는 방식으로, 코드의 역할을 명확히 나누어 관리하기 위해 채택했습니다.

더 자세히 말씀드리자면,
- Model은 게임 내 요소(캐릭터, 아이템 등)를 나타내며, 이들의 논리적 처리와 상태 등을 처리합니다.
- View는 Unity GameObject와 직접 상호작용하며 Model의 값 변화를 반영합니다. UI와 World 요소등의 전환과 애니메이션을 처리합니다.
- Controller는 게임의 상태를 나타내는 State들을 관리하는 역할을 합니다. 이 State들은 입력과 상호작용하며, 게임이 주어진 입력에 어떻게 반응해야 하는지 정의합니다. 입력에 따라 Model, View에 정의해둔 함수를 호출하며 게임을 굴러가게 하는 중간자 역할을 맡습니다.

이러한 구조는 코드의 양을 늘리기도 하지만, 코드의 역할을 정확하게 나눔으로써 각 코드 영역이 해야 할 일을 확실히 하여 협업을 쉽게 합니다. 또한 역할이 명확해지므로 디버깅에도 큰 도움을 줍니다. MVC 패턴을 따라 코드를 작성하는 것은 코드를 별다른 규칙 없이 써내려가는 것보다는 어려웠지만, 협업 및 디버깅 측면에서 상당한 이점을 주었습니다.

#### 사용자 입력 처리
입력의 쉬운 처리와 크로스플랫폼 지원을 고려해 입력을 처리하는 레이어를 구현해 사용했습니다.
Unity가 제공하는 입력 API는 플랫폼에 종속적입니다. 어떤 키가 눌리거나 마우스가 클릭되는 등의 입력을 감지하려 한다면 PC가 아닌 플랫폼에서는 쓰기 어려울 것입니다. 

이를 해결하기 위해 입력을 추상화한 클래스를 만들었고, InputManager를 만들어 입력의 추상화를 관리하도록 했습니다. InputManager는 하드웨어로부터 오는 입력을 추상화하여 State(Controller)에 전달합니다. 이 방법은 State가 추상화된 입력에만 신경쓸 수 있게 만들어, State 코드의 작성 난이도를 낮추고 구조를 간결하게 만듭니다.


#### 액션 기반 처리
게임의 모든 행동은 **액션**으로 정의됩니다. 아이템을 획득하거나, 유물의 효과가 발동되거나, 다음 방에 들어가는 등의 모든 행동을 액션으로 취급하고 처리하도록 구현했습니다. 액션을 처리하면, 그 액션에 해당하는 *파생 액션*들이 생겨납니다. 예를 들어, 스킬을 시전하는 *스킬 시전 액션*은 *피해 액션*과 *자원 소모 액션*을 파생시킵니다.

이 처리 방식은 턴제 기반 전투의 특징을 구현하기 위해 채택했습니다. 플레이어가 가진 유물, 아이템 등 전투에 영향을 줄 수 있는 것들이 있다면, 액션을 처리할 때 파생 액션들을 추가로 생성토록 해 처리하는 것이 좋다고 생각했습니다.


#### 애니메이션의 순차적 처리
게임의 모든 행동을 액션으로 정의했다면, 각 액션에는 애니메이션이 따릅니다. 스킬을 시전한다면 스킬 시전 동작과 피해량이 숫자로 나타나는 애니메이션 등 다양한 애니메이션이 동반됩니다.
실시간 게임과 달리 턴제 게임이므로, 여러 애니메이션의 순차적 처리가 필요한 구조를 고안해야 했습니다. 스킬 시전 동작이 끝나기 전에 피해량을 나타내는 숫자가 떠오른다면 어색하게 느껴질 것입니다. 몇몇 애니메이션은 동시에 처리해야 합니다. 어떤 애니메이션은, 이전에 실행된 몇 개의 애니메이션이 모두 끝날 때까지 기다려야합니다.

이런 복잡한 구조를 처리하려면 그래프가 제일 좋은 구현 방식이라 생각했습니다. 특히, 비순환 유향 그래프(Directed Acyclic Graph, DAG)가 애니메이션의 선후관계를 표현하기 가장 좋아보였는데, 여러 애니메이션이 복잡하게 얽힌 관계를 DAG의 구조로 표현할 수 있기 때문이었습니다. 
그러나 DAG를 이용해서 애니메이션을 구현하는 것은 좋은 방법이 아니라 판단했습니다. 선행되는 애니메이션의 참조를 모두 알고 있어야 했고, 각 애니메이션이 어떤 애니메이션을 기다려야 할지 알아내는 것도 어려웠기 때문에 애니메이션 DAG를 만드는 것이 아주 어렵고 복잡하다고 결론지었기 때문입니다. 

최종적으로 채택한 구조는 트리(Tree) 구조였습니다. 액션이 파생 액션을 생성하는 것은 트리의 노드에 자식 노드를 추가하는 것과 같다고 생각했고, DAG보다는 제한적이지만 선후관계를 표현하는 것도 가능하기 때문이었습니다. 게다가 트리는 상대적으로 생성하기도 쉬우며 직관적입니다.
자식 노드를 2차원 배열로 관리하여, children[0]에 있는 자식 애니메이션이 모두 처리되면 children[1]의 애니메이션들을 처리하는 방식으로 관리하면 애니메이션의 선후관계를 잘 정리하면서도, 병렬적으로 재생해야하는 애니메이션도 쉽게 처리할 수 있었습니다.
